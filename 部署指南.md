# 部署指南

本文按当前仓库真实代码从 0 到 1 写部署流程。核心结论：`agent-master` 可以部署在任意 namespace；它只需要有权限调度 `agent-runtime` namespace 里的 Agent，并且能通过 Kubernetes Service DNS 代理访问这些 Agent。

下文使用两个占位：

- `{masterNamespace}`：`agent-master` 自身部署位置，由平台决定。
- `{runtimeNamespace}`：Agent 工作负载目标 namespace，当前建议为 `agent-runtime`，对应 `config.yaml` 的 `kubernetes.namespace`。

## 1. 服务边界

`agent-master` 自身是 Bun + TypeScript + Fastify 服务，Dockerfile 中启动命令为：

```bash
bun src/server.ts
```

对外接口：

- `GET /health`：健康检查。
- `/runtime`：创建、查询、重启、删除当前用户 Agent。
- `GET /runtime/events`：平台控制面 SSE。
- `/agent/*`：代理到当前用户 Agent Service，转发时去掉 `/agent` 前缀。

`agent-master` 创建的 Agent 工作负载固定为：

```text
Deployment 1 副本 + Service
```

代理访问使用完整集群 DNS：

```text
http://{serviceName}.{runtimeNamespace}.svc.cluster.local:{runtime.port}{path}
```

所以 `agent-master` 不必和 Agent 在同一个 namespace，但必须满足三点：

1. ServiceAccount 有权限管理 `{runtimeNamespace}` 内的 Deployment、Service、Pod 等资源。
2. 集群网络允许 `{masterNamespace}` 访问 `{runtimeNamespace}` 的 Service。
3. `config.yaml` 中 `kubernetes.namespace` 与实际 `{runtimeNamespace}` 一致。

## 2. 从 0 到 1 部署顺序

1. 选择 `{masterNamespace}` 和 `{runtimeNamespace}`。
2. 创建 `{runtimeNamespace}`，用于承载 Redis 和 Agent 工作负载；Redis 也可以是外部服务。
3. 在 `{runtimeNamespace}` 准备用户工作目录和 `resources/templates` 初始化模板。
4. 构建并推送 `agent-runtime` 镜像。
5. 本地执行 `bun test`、`bun run typecheck`。
6. 构建并推送 `agent-master` 镜像。
7. 在 `{masterNamespace}` 创建 `agent-master` 的 ConfigMap、Secret、Deployment、Service。
8. 创建跨 namespace RBAC：Role 在 `{runtimeNamespace}`，RoleBinding 绑定 `{masterNamespace}` 中的 ServiceAccount。
9. 验证 `GET /health`。
10. 调用 `POST /runtime` 创建 Agent。
11. 调用 `/agent/*` 验证代理链路。

## 3. 配置关系

`agent-master` 运行时读取两类配置：

1. 环境变量：`PORT`、`HOST`、`LOG_LEVEL`、`K8S_CLUSTERS`，用于 HTTP 服务监听。
2. `/app/config.yaml`，用于 Redis、Agent 镜像、用户工作目录初始化根、Runtime PVC/subPath、Kubernetes 目标 namespace。

关键配置示例：

```yaml
server:
  port: 3000
  host: 0.0.0.0
  log: info
redis:
  host: redis.agent-runtime.svc.cluster.local
  port: 6379
  db: 0
  key: agent-runtime:user
  password: ""
runtime:
  image: agent-image:latest
  ttl: 3600
  port: 4096
  workdir: /nas/agent-master/users
  workspacePvcClaimName: agent-runtime-nas
  workspacePvcSubPathRoot: agent-master/users
init:
  templatesRoot: ./resources/templates
kubernetes:
  cluster: default
  namespace: agent-runtime
  clusters:
    - name: default
      namespace: agent-runtime
      auth: inCluster
      scheduling:
        enabled: true
        maxRuntime: 100
```

注意：

- `kubernetes.namespace` 是 Agent 工作负载目标 namespace，不是 `agent-master` 自身必须所在 namespace。
- `redis.host` 如果 Redis 不在 `{masterNamespace}`，建议写完整 DNS。
- `maxRuntime` 是代码识别的字段名，不能改成其他名字。
- 密码不要明文写入仓库，使用 `${REDIS_PASSWORD}` 并通过 Secret 注入。

## 4. 工作目录与初始化模板

 当前约定：

```text
Runtime Deployment 使用单个 PVC，两个 subPath 挂载，禁止 hostPath：
PVC {runtime.workspacePvcClaimName} subPath {runtime.workspacePvcSubPathRoot}/{userId}/runtime -> /app
PVC {runtime.workspacePvcClaimName} subPath {runtime.workspacePvcSubPathRoot}/{userId}/global -> ~

对应目录结构：
{runtime.workdir}/{userId}/
├── runtime/                      # 用户项目工作目录 → /app
│   ├── AGENTS.md
│   └── .opencode/
└── global/                       # OpenCode 全局数据 → ~
    └── .config/opencode/         # → ~/.config/opencode
    └── .local/share/opencode/   # → ~/.local/share/opencode
    └── .cache/opencode/         # → ~/.cache/opencode
```

创建 Agent 前必须确保：

- `{runtime.workdir}/{userId}` 可创建。
- `{runtime.workdir}/{userId}/runtime/`、`{runtime.workdir}/{userId}/runtime/.opencode/` 可创建。
- `{runtime.workdir}/{userId}/global/.config/opencode/`、`{runtime.workdir}/{userId}/global/.local/share/opencode/`、`{runtime.workdir}/{userId}/global/.cache/opencode/` 可创建，用于持久化 OpenCode 用户级配置和数据。
- `resources/templates` 初始化模板目录存在 `AGENTS.md` 和 `opencode.json`。
- `agent-master` 的 `runtime.workdir` 和后续 Agent 工作负载使用的 PVC 指向同一套存储根路径。

Runtime Deployment 禁止使用 `hostPath`；本地测试和生产环境都必须准备 PVC，并通过 `runtime.workspacePvcClaimName` + `runtime.workspacePvcSubPathRoot` 生成两个 subPath 挂载。

## 5. RBAC

如果 `agent-master` 部署在 `{masterNamespace}`，但调度 `{runtimeNamespace}`，RBAC 关系应这样建：

- ServiceAccount：在 `{masterNamespace}`。
- Role：在 `{runtimeNamespace}`。
- RoleBinding：在 `{runtimeNamespace}`，subject 指向 `{masterNamespace}` 的 ServiceAccount。

示例：

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: agent-master
  namespace: {masterNamespace}
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: agent-master-runtime-manager
  namespace: agent-runtime
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "patch", "update", "delete"]
  - apiGroups: [""]
    resources: ["services"]
    verbs: ["get", "list", "watch", "create", "patch", "update", "delete"]
  - apiGroups: [""]
    resources: ["pods", "events", "resourcequotas", "limitranges"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: agent-master-runtime-manager
  namespace: agent-runtime
subjects:
  - kind: ServiceAccount
    name: agent-master
    namespace: {masterNamespace}
roleRef:
  kind: Role
  name: agent-master-runtime-manager
  apiGroup: rbac.authorization.k8s.io
```

当前代码会读取 Node 做调度检查。如集群启用该能力，还需要给 ServiceAccount 绑定 `nodes get/list/watch` 的 ClusterRole。

## 6. Deployment 要点

`agent-master` Deployment 可以在 `{masterNamespace}`：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agent-master
  namespace: {masterNamespace}
spec:
  replicas: 1
  selector:
    matchLabels:
      app: agent-master
  template:
    metadata:
      labels:
        app: agent-master
    spec:
      serviceAccountName: agent-master
      containers:
        - name: agent-master
          image: ghcr.io/archaiharness/agent-master:<version>
          ports:
            - name: http
              containerPort: 3000
          env:
            - name: PORT
              value: "3000"
            - name: HOST
              value: 0.0.0.0
            - name: LOG_LEVEL
              value: info
          volumeMounts:
            - name: config
              mountPath: /app/config.yaml
              subPath: config.yaml
            - name: nas
              mountPath: /nas
          readinessProbe:
            httpGet:
              path: /health
              port: http
          livenessProbe:
            httpGet:
              path: /health
              port: http
      volumes:
        - name: config
          configMap:
            name: agent-master-config
        - name: nas
          persistentVolumeClaim:
            claimName: agent-master-nas
```

Service 同样放在 `{masterNamespace}`，供网关或端口转发访问。

## 7. 部署后验证

端口转发：

```bash
kubectl -n {masterNamespace} port-forward svc/agent-master 3000:3000
```

健康检查：

```bash
curl http://127.0.0.1:3000/health
```

创建 Agent：

```bash
curl -X POST http://127.0.0.1:3000/runtime \
  -H 'x-user-id: user-a'
```

检查 Agent 工作负载是否出现在 `{runtimeNamespace}`：

```bash
kubectl -n {runtimeNamespace} get deploy -l app=opencode-runtime
kubectl -n {runtimeNamespace} get svc -l app=opencode-runtime
kubectl -n {runtimeNamespace} get pod -l app=opencode-runtime
```

验证代理：

```bash
curl http://127.0.0.1:3000/agent/project/current \
  -H 'x-user-id: user-a'
```

创建会话：

```bash
curl -X POST 'http://127.0.0.1:3000/agent/session?directory=/app/workspace-a' \
  -H 'x-user-id: user-a' \
  -H 'Content-Type: application/json' \
  -d '{"title":"smoke test"}'
```

`directory` 是 OpenCode 官方 query 参数，由调用方显式传入。`agent-master` 只去掉 `/agent` 前缀并透明透传，不读取请求体 `scene`，也不做 `scene -> directory` 转换。

## 8. GitHub CI/CD 流程

建议拆成两个 workflow：

1. `ci.yml`：PR 和 main push 触发，只做验证和镜像构建检查。
2. `deploy.yml`：main push 或手动触发，构建、推送镜像、部署到 Kubernetes。

### 8.1 CI 步骤

`ci.yml` 建议步骤：

1. Checkout。
2. Setup Bun。
3. `bun install --frozen-lockfile`。
4. `bun test`。
5. `bun run typecheck`。
6. Docker build，但不 push。
7. 扫描文档和代码中是否残留已废弃的服务名和接口前缀。

### 8.2 CD 步骤

`deploy.yml` 建议步骤：

1. Checkout。
2. Setup Bun。
3. `bun install --frozen-lockfile`。
4. `bun test`。
5. `bun run typecheck`。
6. Login GHCR。
7. Build `ghcr.io/archaiharness/agent-master:${{ github.sha }}`。
8. Push 镜像。
9. 根据 GitHub Environment 选择 kubeconfig、`{masterNamespace}`、`{runtimeNamespace}`。
10. `kubectl apply` namespace、RBAC、ConfigMap、Deployment、Service。
11. `kubectl rollout status deploy/agent-master -n {masterNamespace}`。
12. 端口转发或通过测试网关验证 `GET /health`。
13. 可选 smoke test：`POST /runtime` + `GET /agent/project/current`。

### 8.3 GitHub Secrets / Variables

建议使用 GitHub Environments 管理不同环境。

Secrets：

- `GHCR_TOKEN`：推送 GHCR 镜像。
- `KUBE_CONFIG`：目标集群 kubeconfig。
- `REDIS_PASSWORD`：如 Redis 启用 AUTH。

Variables：

- `MASTER_NAMESPACE`：`agent-master` 自身部署位置。
- `RUNTIME_NAMESPACE`：Agent 工作负载目标 namespace，例如 `agent-runtime`。
- `AGENT_RUNTIME_IMAGE`：Agent 镜像。
- `REDIS_HOST`：Redis 服务地址。

### 8.4 deploy.yml 骨架

```yaml
name: deploy

on:
  workflow_dispatch:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: bun install --frozen-lockfile
      - run: bun test
      - run: bun run typecheck
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GHCR_TOKEN }}
      - run: |
          docker build -t ghcr.io/archaiharness/agent-master:${{ github.sha }} .
          docker push ghcr.io/archaiharness/agent-master:${{ github.sha }}
      - run: |
          mkdir -p ~/.kube
          echo "${{ secrets.KUBE_CONFIG }}" > ~/.kube/config
          kubectl create namespace "${{ vars.RUNTIME_NAMESPACE }}" --dry-run=client -o yaml | kubectl apply -f -
          kubectl create namespace "${{ vars.MASTER_NAMESPACE }}" --dry-run=client -o yaml | kubectl apply -f -
          kubectl -n "${{ vars.MASTER_NAMESPACE }}" set image deployment/agent-master agent-master=ghcr.io/archaiharness/agent-master:${{ github.sha }}
          kubectl -n "${{ vars.MASTER_NAMESPACE }}" rollout status deployment/agent-master --timeout=180s
```

首次部署时不能只 `set image`，必须先 `kubectl apply` RBAC、ConfigMap、Deployment、Service。建议将这些 YAML 放到 `deploy/k8s/`，用变量渲染后再 apply。

## 9. 发布前检查

发布前必须确认：

- `bun test` 通过。
- `bun run typecheck` 通过。
- `GET /health` 返回 `agent-master` 和 `ok`。
- `POST /runtime` 能在 `{runtimeNamespace}` 创建 Agent Deployment 和 Service。
- `/agent/*` 能代理到 Agent Service。
- ConfigMap 不含真实 Token、Cookie、密码、证书、私钥。
- 上游网关注入可信 `x-user-id`。
- `Authorization`、`proxy-authorization` 和 hop-by-hop Header 不透传到 Agent。
