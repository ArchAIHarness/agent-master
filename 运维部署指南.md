# 运维部署指南

本文面向运维、平台工程和发布负责人，说明如何从 0 到 1 部署 `agent-master`，如何维护 `agent-runtime` 镜像，以及如何设计 GitHub CI/CD。本文只提供 CI/CD 建议，**当前仓库不新增 `.github/workflows` 文件**。

## 1. 服务边界

`agent-master` 是 Agent Runtime 控制面服务，负责用户 Runtime 生命周期、Redis 状态、Kubernetes Deployment + Service 创建、Runtime 原生 API 代理，以及 WebUI Interaction BFF。

`agent-runtime` 是被 `agent-master` 拉起的运行时镜像。当前镜像内实现为 OpenCode Web Runtime，但运维上应按“Agent Runtime 镜像”治理，不把 OpenCode 当成 `agent-master` 的核心边界。

## 2. 从 0 到 1 部署顺序

1. 准备 Kubernetes 集群和目标 Namespace。
2. 准备 Redis 服务。
3. 准备用户工作目录根路径与 scene 配置源目录。
4. 构建并推送 `agent-runtime` 镜像。
5. 构建并推送 `agent-master` 镜像。
6. 创建 `agent-master` ConfigMap 与必要 Secret。
7. 创建 ServiceAccount、RBAC、Deployment、Service。
8. 接入上游网关，确保上游鉴权后注入 `x-user-id`。
9. 验证健康检查、Runtime 创建、会话创建、消息、SSE、文件和附件能力。

## 3. Kubernetes 基础资源

### 3.1 Namespace

```bash
kubectl create namespace agent-runtime
```

`config.yaml` 中 `kubernetes.namespace` 必须与实际部署 Namespace 一致。

### 3.2 ServiceAccount 与 RBAC

`agent-master` 需要在目标 Namespace 管理 Runtime Deployment 与 Service，并读取调度状态。最小权限建议：

- `deployments`：get、list、watch、create、patch、update、delete。
- `services`：get、list、watch、create、patch、update、delete。
- `pods`：get、list、watch。
- `events`：get、list、watch。
- `resourcequotas`：get、list。
- `limitranges`：get、list。
- `nodes`：get、list；如集群策略不允许读取 Node，需要由平台侧提供等效资源检查能力。

示例骨架：

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: agent-master
  namespace: agent-runtime
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: agent-master-runtime-manager
  namespace: agent-runtime
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "patch", "update", "delete"]
  - apiGroups: [""]
    resources: ["services", "pods", "events", "resourcequotas", "limitranges"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["services"]
    verbs: ["create", "patch", "update", "delete"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: agent-master-runtime-manager
  namespace: agent-runtime
subjects:
  - kind: ServiceAccount
    name: agent-master
    namespace: agent-runtime
roleRef:
  kind: Role
  name: agent-master-runtime-manager
  apiGroup: rbac.authorization.k8s.io
```

如需读取 Node，按集群安全策略额外配置 `ClusterRole` / `ClusterRoleBinding`。

## 4. Redis 准备

`agent-master` 使用 Redis 保存 `userId -> Runtime` 映射、状态和 TTL。

要求：

- `agent-master` Pod 内可访问。
- 支持 DB 选择和 TTL。
- 如启用 AUTH，密码通过 Kubernetes Secret 或受控环境变量注入。
- 不把 Redis 密码写入仓库、镜像或普通日志。

配置示例：

```yaml
redis:
  host: redis
  port: 6379
  db: 0
  key: agent-runtime:user
  password: ""
```

如需密码：

```yaml
redis:
  password: ${REDIS_PASSWORD}
```

并通过 Secret 注入 `REDIS_PASSWORD`。

## 5. 存储目录准备

### 5.1 用户工作目录

`runtime.workdir` 是用户工作目录根路径。Runtime 创建时按用户拼接：

```text
{runtime.workdir}/{userId} -> /app
```

用户目录承载用户工作文件、默认 `AGENTS.md`、默认 `.opencode/`、Runtime 运行态持久化目录 `.runtime/` 和各 scene 工作目录。

### 5.2 Runtime 持久化目录

当前 `agent-runtime` 镜像需要持久化：

```text
{runtime.workdir}/{userId}/.runtime/opencode/share  -> /root/.local/share/opencode
{runtime.workdir}/{userId}/.runtime/opencode/config -> /root/.config/opencode
```

这些目录可能包含会话数据库、WAL、日志、插件依赖、模型配置或 Token。备份、迁移、排障时按敏感数据处理，不写入公开文档、API 响应或普通日志。

### 5.3 scene 配置源目录

每个 scene 源目录必须包含：

```text
{runtime.scenes.<scene>}/AGENTS.md
{runtime.scenes.<scene>}/.opencode/
```

Runtime 创建时：

```text
{runtime.scenes.<scene>} -> initContainer /scene-config/{scene}
/scene-config/{scene}/AGENTS.md   -> /app/{scene}/AGENTS.md
/scene-config/{scene}/.opencode/. -> /app/{scene}/.opencode/
```

主 Runtime 容器只挂载用户工作目录 `/app`，不直接挂载 scene 源目录。

## 6. agent-runtime 镜像维护

### 6.1 镜像必须满足

当前 Runtime 启动命令由 `agent-master` 生成：

```sh
opencode web --port 4096 --hostname 0.0.0.0
```

镜像要求：

- 存在 `opencode` 命令。
- 监听 `0.0.0.0:{runtime.port}`，默认 `4096`。
- 存在 `/bin/sh`、`sleep`、`base64`。
- 能读写 `/app`、`/root/.local/share/opencode`、`/root/.config/opencode`。
- 支持优雅退出，避免运行态数据库和配置写入中断。

### 6.2 推荐镜像结构

```dockerfile
FROM node:22-slim

RUN apt-get update \
  && apt-get install -y --no-install-recommends ca-certificates git openssh-client bash coreutils \
  && rm -rf /var/lib/apt/lists/*

# 安装当前 Runtime 实现所需 CLI。
# 示例：RUN npm install -g <runtime-cli-package>

WORKDIR /app
EXPOSE 4096
CMD ["opencode", "web", "--port", "4096", "--hostname", "0.0.0.0"]
```

### 6.3 镜像验证

```bash
docker run --rm <agent-runtime-image> opencode --version
docker run --rm <agent-runtime-image> /bin/sh -c 'command -v base64 && command -v sleep'
```

上线前还要在临时 Namespace 验证 Runtime 健康检查、会话创建、异步消息、SSE、文件列表、文件内容、附件写入和 Runtime 重启后的会话恢复。

### 6.4 版本策略

推荐：

- 不可变标签：`ghcr.io/<org>/agent-runtime:<semver-or-sha>`。
- 环境标签：`dev`、`staging`、`prod`。
- 生产环境避免只使用 `latest`。

`config.yaml` 中 `runtime.image` 应指向明确版本。

## 7. agent-master 镜像构建与部署

### 7.1 构建前验证

```bash
bun install --frozen-lockfile
bun test
bun run typecheck
```

### 7.2 构建镜像

当前 Dockerfile 基于 `oven/bun:1.3.13-slim`，服务监听端口为 `3000`。

```bash
docker build -t ghcr.io/<org>/agent-master:<version> .
docker push ghcr.io/<org>/agent-master:<version>
```

### 7.3 ConfigMap

将 `config.yaml` 挂载到 `/app/config.yaml`。

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: agent-master-config
  namespace: agent-runtime
data:
  config.yaml: |
    server:
      port: 3000
      host: 0.0.0.0
      log: info
    redis:
      host: redis
      port: 6379
      db: 0
      key: agent-runtime:user
      password: ""
    runtime:
      image: ghcr.io/<org>/agent-runtime:<version>
      ttl: 3600
      timeout: 60000
      port: 4096
      workdir: /nas/agent-master/users
      scenes:
        coding: /nas/agent-master/scenes/coding
        review: /nas/agent-master/scenes/review
    nas:
      path: /nas/agent-master
    kubernetes:
      cluster: default
      namespace: agent-runtime
      clusters:
        - name: default
          namespace: agent-runtime
          auth: inCluster
          scheduling:
            enabled: true
            maxRuntime: 100
```

### 7.4 Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: agent-master
  namespace: agent-runtime
spec:
  replicas: 1
  selector:
    matchLabels:
      app: agent-master
  template:
    metadata:
      labels:
        app: agent-master
    spec:
      serviceAccountName: agent-master
      containers:
        - name: agent-master
          image: ghcr.io/<org>/agent-master:<version>
          ports:
            - name: http
              containerPort: 3000
          env:
            - name: REDIS_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: agent-master-secret
                  key: redis-password
                  optional: true
          volumeMounts:
            - name: config
              mountPath: /app/config.yaml
              subPath: config.yaml
          readinessProbe:
            httpGet:
              path: /api/v1/health
              port: http
          livenessProbe:
            httpGet:
              path: /api/v1/health
              port: http
      volumes:
        - name: config
          configMap:
            name: agent-master-config
```

### 7.5 Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: agent-master
  namespace: agent-runtime
spec:
  selector:
    app: agent-master
  ports:
    - name: http
      port: 3000
      targetPort: http
```

## 8. 配置字段说明

| 字段 | 必填 | 说明 |
|---|---:|---|
| `server.port` | 是 | HTTP 服务端口，默认 `3000`。 |
| `server.host` | 是 | 监听地址，容器内通常为 `0.0.0.0`。 |
| `server.log` | 是 | 日志级别。 |
| `redis.host` | 是 | Redis 服务名或主机名。 |
| `redis.port` | 是 | Redis 端口。 |
| `redis.db` | 是 | Redis DB 编号。 |
| `redis.key` | 是 | Runtime 映射 Key 前缀。 |
| `redis.password` | 否 | Redis 密码；真实值通过 Secret 或环境变量注入。 |
| `runtime.image` | 是 | `agent-runtime` 镜像。 |
| `runtime.ttl` | 是 | Runtime 租约秒数。 |
| `runtime.timeout` | 是 | Runtime API 代理超时毫秒数。 |
| `runtime.port` | 是 | Runtime 容器端口，默认 `4096`。 |
| `runtime.workdir` | 是 | 用户工作目录根路径。 |
| `runtime.scenes` | 是 | 预设 scene 配置源目录映射。 |
| `nas.path` | 是 | NAS 根路径说明字段。 |
| `kubernetes.cluster` | 是 | 默认集群逻辑名。 |
| `kubernetes.namespace` | 是 | Runtime 目标 Namespace。 |
| `kubernetes.clusters[]` | 是 | 可调度集群配置。 |
| `kubernetes.clusters[].scheduling.maxRuntime` | 是 | 单 Namespace Runtime 数量上限。 |

## 9. 上游网关要求

`agent-master` 不解析公网用户 Token。上游必须完成鉴权，并注入：

```http
x-user-id: <user-ref>
```

要求：

- `x-user-id` 必须是可信上游注入值。
- 公网客户端不能绕过网关直连 `agent-master`。
- `Authorization` 如进入 `agent-master`，不得透传到 Runtime。
- 上游应限制请求体大小、SSE 连接数量和异常重试频率。

## 10. 日常运维

### 10.1 健康检查

```bash
kubectl -n agent-runtime get deploy agent-master
kubectl -n agent-runtime get pod -l app=agent-master
kubectl -n agent-runtime logs deploy/agent-master
```

服务健康接口：

```http
GET /api/v1/health
```

### 10.2 Runtime 资源检查

```bash
kubectl -n agent-runtime get deploy -l app=opencode-runtime
kubectl -n agent-runtime get svc -l app=opencode-runtime
kubectl -n agent-runtime get pod -l app=opencode-runtime
```

如果后续更换 Runtime 实现，label 命名应随实现同步治理，避免监控误判。

### 10.3 重启与回滚

- `agent-master` 配置变更后，滚动重启 `agent-master` Deployment。
- `runtime.image` 更新只影响后续新建 Runtime；已有 Runtime 是否重建由运维策略决定。
- Runtime 配置损坏时，当前启动 wrapper 会隔离 `/root/.config/opencode` 中的异常配置到 `.quarantine` 后用最小配置重试。
- `.quarantine` 是敏感现场，不上传到公开系统。

### 10.4 常见故障

| 现象 | 排查方向 |
|---|---|
| Runtime 创建失败 | RBAC、Namespace、ResourceQuota、LimitRange、Node 状态、镜像拉取权限。 |
| Runtime Service 不可访问 | Service selector、Pod readiness、端口 `4096`、NetworkPolicy。 |
| 会话恢复失败 | 用户工作目录、`.runtime` 持久化目录、Runtime 镜像版本。 |
| 附件上传失败 | Runtime shell、`base64`、`/app/.interaction/uploads` 写权限。 |
| Redis 状态异常 | Redis 连接、DB、Key 前缀、TTL、AUTH。 |

## 11. GitHub CI/CD 配置建议

当前仓库不新增 `.github/workflows`。如果后续要配置 GitHub Actions，建议拆成两个 workflow。

### 11.1 agent-master CI

触发条件：

- Pull Request。
- main 分支 push。
- 手动 workflow_dispatch。

步骤建议：

1. Checkout。
2. Setup Bun。
3. `bun install --frozen-lockfile`。
4. `bun test`。
5. `bun run typecheck`。
6. 敏感信息扫描。
7. Docker build。
8. 推送到 GHCR 或企业镜像仓库。
9. 可选：生成 SBOM 和镜像漏洞扫描。

### 11.2 agent-runtime CI

触发条件：

- Runtime Dockerfile 或 runtime 依赖变更。
- 手动发布。

步骤建议：

1. Checkout。
2. Docker build。
3. 验证 `opencode --version`。
4. 验证 `/bin/sh`、`base64`、`sleep` 存在。
5. 启动容器并检查端口监听。
6. 推送不可变版本标签。
7. 推送环境标签前要求人工审批。

### 11.3 GitHub Secrets 建议

需要的 Secret 只在 GitHub 仓库或组织配置，不写入代码：

| Secret | 用途 |
|---|---|
| `GHCR_TOKEN` | 推送 GHCR 镜像；如使用 `GITHUB_TOKEN` 可不单独配置。 |
| `KUBE_CONFIG_STAGING` | 可选，部署 staging。 |
| `KUBE_CONFIG_PROD` | 可选，部署 production，必须加环境保护。 |
| `REDIS_PASSWORD_STAGING` | 可选，生成或校验 staging Secret。 |
| `REDIS_PASSWORD_PROD` | 可选，生成或校验 production Secret。 |

建议启用：

- GitHub Environments：`staging`、`production`。
- production 人工审批。
- 分支保护。
- 镜像签名和 SBOM。
- Dependabot 或等效依赖升级机制。

## 12. 发布检查清单

发布前必须确认：

- `bun test` 通过。
- `bun run typecheck` 通过。
- 镜像构建成功。
- ConfigMap 不含真实 Token、Cookie、密码、证书、私钥。
- Secret 由受控系统注入。
- `runtime.image` 指向明确版本。
- Redis 可访问且 TTL 生效。
- ServiceAccount 权限满足 Runtime 创建与回收。
- 用户工作目录和 scene 配置源目录存在。
- 上游网关注入 `x-user-id`。
- 健康检查、Runtime 创建、会话创建、异步消息、SSE、附件、文件接口完成验收。

## 13. 安全边界

禁止事项：

- 不把 kubeconfig、Token、Cookie、密码、证书、私钥写入仓库。
- 不把 Runtime 持久化目录暴露给公开 API。
- 不把 `.runtime`、`.quarantine`、运行日志上传到公开系统。
- 不让公网客户端绕过上游直接伪造 `x-user-id`。
- 不在生产环境长期使用 `latest` 作为唯一镜像版本。
- 不在当前仓库直接新增 GitHub workflow；如需启用 CI/CD，应先按本文建议评审后再单独落地。
